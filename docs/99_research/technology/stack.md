# 技術構成

本ドキュメントは採用技術とその選定理由をまとめた資料である。
詳細な外部連携仕様は `02_specification/integrations/` を参照。


## 構成図

```
サロン側
  └── Gmail（専用）         ← サロンボードからの口コミ通知メール

開発者管理（クラウド）
  ├── GAS                   ← メール監視・本文抽出・AI呼び出し
  ├── Gemini API            ← 返信文生成・ブログ原稿生成（Vision含む）
  ├── Next.js（Vercel）     ← Webアプリ（確認・編集・コピーUI）
  ├── Supabase              ← DB（PostgreSQL）+ Auth（フェーズ1.5〜）
  └── Cloudflare R2         ← 画像ストレージ（フェーズ2〜）
```


## 技術スタック詳細

### フロントエンド

| 項目 | 採用技術 | 備考 |
|---|---|---|
| フレームワーク | Next.js（App Router） | |
| ホスティング | Vercel | フェーズ1はHobbyプラン（無料） |
| UIコンポーネント | shadcn/ui | |
| スタイリング | Tailwind CSS | |

### バックエンド・API

Next.jsのAPI Routeで実装。自前サーバーは持たない。

### 認証

| フェーズ | 対応 |
|---|---|
| フェーズ1 | 認証なし（1サロン固定） |
| フェーズ1.5〜 | Supabase Auth（メール+パスワード）|

### データベース

| 項目 | 採用技術 | 備考 |
|---|---|---|
| DB | Supabase（PostgreSQL） | |
| RLS | 全テーブルに設定 | salon_id / store_id でデータ分離 |
| 自動削除 | pg_cron | expires_at を基準に30日で削除 |

### ストレージ

| 項目 | 採用技術 | 備考 |
|---|---|---|
| 画像保存 | Cloudflare R2 | フェーズ2〜（ブログ画像） |
| 接続方法 | @aws-sdk/client-s3 | R2はS3互換APIのため |
| 無料枠 | 10GB | 複数サロン展開でも当面無料枠内の想定 |

Supabase Storageを使わない理由: 無料枠が1GBと小さく、複数サロン・複数枚投稿を考慮するとすぐ上限に達する。

### AI

| 項目 | 採用技術 | 備考 |
|---|---|---|
| モデル | Gemini 2.0 Flash | コスト・速度・品質のバランスが最適 |
| 用途（フェーズ1） | テキスト生成 | 口コミ返信文の生成 |
| 用途（フェーズ2） | Vision + テキスト生成 | ヘアスタイル写真→ブログ原稿生成 |
| コスト | 入力$0.10 / 出力$0.40（100万トークンあたり） | 1サロン月数十円程度 |

### 自動化（メール監視）

| 項目 | 採用技術 | 備考 |
|---|---|---|
| ツール | Google Apps Script（GAS） | |
| トリガー | 時間駆動型（数分おきにメールを巡回） | |
| 認証 | OAuth（サロンのGmailと連携） | サロンごとに初回セットアップが必要 |
| 制限 | 1回の実行6分・1日の合計90分 | 今回のユースケースでは問題なし |
| 将来の移行先 | Vercel API Route | 10サロン超えたあたりで移行を検討 |

GASを選んだ理由: Gmailとのネイティブ連携が最もシンプル。無料。メール監視→AI API呼び出し程度の処理なら制限に引っかからない。n8n・Makeはオーバースペック。

## 採用しなかった技術

| 技術 | 不採用理由 |
|---|---|
| 自前サーバー | Vercel + GASで完結するため不要。運用コスト・保守負担が増えるだけ |
| n8n / Make | Gmailとの連携がGASより複雑。有料プランが必要なケースがある |
| Supabase Storage | 無料枠1GBが小さすぎる。複数サロン・複数枚投稿に対応できない |
| LINE Messaging API | サロンボードのメール通知で代替可能。オーナーへの重複通知になるため不採用 |
| Playwright自動操作 | HPB利用規約【2】に抵触するため不採用 |
| Neon | DBとしての性能はNeonが優秀だが、Auth・DB・Storageを一元管理したいためSupabaseを選択。Storageが不要な構成ならNeonの方がVercelとの統合・ブランチ機能の面で優れている |
| AWS | EC2・RDS・S3・IAM・VPC・ALBと設定項目が膨大。1人でスモールスタートする規模に対してオーバースペック |
| GCP（フルスタック） | Cloud Run + Cloud SQL + Vertex AIという構成も検討したが、設定・運用コストが高い。Gemini APIはGCP外からも呼び出せるため、GCPに統一するメリットがない |
| ローカルLLM | 日本語品質がGemini Flashより劣る。GPUサーバーの運用コストがAPI従量課金より高くなる。データを外部に出せない制約がある場合に有効だが、口コミ返信生成においてその制約はない |


## 将来の拡張ポイント

- **GAS → Vercel API Route移行**: サロン数が10を超えたあたりで検討。複数サロンのOAuth管理が複雑になるため
- **Gemini → モデル変更**: プロンプト層を抽象化しておくことでモデル変更を容易にする
- **Supabase → 別DBへの移行**: 当面はSupabaseで十分。スケールに応じて検討
