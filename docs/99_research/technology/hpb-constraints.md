# HPB 制約・自動化検討メモ

**作成日**: 2026/4/1 | A案採用の根拠と検討過程


## 1. HPBの構造的制約

- HPBはサロン向けAPIを公開していない（口コミ返信・投稿のAPIは存在しない）
- 口コミ返信・投稿はサロンボード（Web管理画面）上でのみ可能
- お客さんの口コミ投稿もHPBアプリ/Webからのみ可能
- HPBのWebページから口コミ通知メールをサロン側のメールアドレスに送る設定が可能


## 2. 規約上の制約

**参照URL**: https://beauty.hotpepper.jp/doc/terms/terms.html

Playwrightなどによるブラウザ自動操作・自動投稿が抵触する条項：

> 【2】当社の承認した以外の方法により本サービスを利用する行為

違反した場合、利用停止・ID剥奪・損害賠償請求等の措置が取られる可能性がある。

**機能B（お客さんの口コミ自動投稿）についても同様。**
技術的に不可能なことに加え、サロンが口コミ投稿フローを業務に組み込んで誘導する行為は
【3】本サービスを商用目的で利用する行為 にも抵触する余地がある。

サロンボード / 口コミ通知メールから確認できる情報：
- 投稿者名（例: あゆさん）
- 性別・年代（例: 女性/20代後半）
- 投稿日
- 総合評価・詳細評価（雰囲気・接客・技術・メニュー）
- 口コミ本文
- クーポン名・施術メニュー


## 4. 検討した自動化パターンと評価

| 方式 | 手動アクション | 規約リスク | 採用 |
|------|-------------|-----------|------|
| A. オーナーがコピペ + Webアプリ | 1回（ペーストのみ） | なし | ✅ 採用 |
| B. Chrome拡張（返信欄に挿入） | 1回（送信のみ） | グレー | 将来検討 |
| C. Playwright自動操作・自動投稿 | 0回 | 高（規約違反）| ❌ 不採用 |

メール監視による口コミ自動取り込みも当初検討したが、メール本文に含まれる投稿者情報を機械処理することの個人情報リスクと、HPB側のメール仕様変更リスクを踏まえ見送った。オーナーが必要な口コミだけをコピーして貼り付ける運用に統一する。


## 5. 採用アーキテクチャ（A案）

```
口コミ投稿
  ↓
オーナーがサロンボードで口コミを確認し、本文をコピーする
  ↓
Webアプリ（Next.js）に貼り付け、担当スタッフを選択する
  ↓
Server Action が Gemini API を呼び出し、スタッフのトーン例文を few-shot として返信文を生成する
  ↓
Supabase に返信履歴として保存する
  ↓
オーナーが返信文をコピー → サロンボードにペーストして投稿（手動1アクション）
```


## 6. 個人情報の取り扱い方針

### フェーズ1（現在）

口コミ本文・投稿者情報は保存しない。生成した返信文のみSupabaseに保存する。

理由: 「あゆさん（女性/20代後半）」はサロン側が誰か特定できるケース（常連客等）があり、個人情報に該当しうる。スモールスタートの段階でリーガルチェックのコストをかけないための判断。

### フェーズ2以降（SaaS本格展開時）

口コミ情報を保存する構成に拡張する。このタイミングでリーガルチェックを入れ、プライバシーポリシーを整備する。法的根拠は「サロンからの業務委託」として位置づける。


## 7. 未解決・将来検討

- サロンボード側の仕様変更（UI変更によるコピーしづらさ等）リスクは継続監視が必要
