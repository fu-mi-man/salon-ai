# review-summary

PR に付いた全てのレビューコメント（CodeRabbit、code-review プラグイン、人間のレビューなど）を漏れなく収集し、指摘事項をまとめて修正方針を提案する。

## なぜこのスキルが必要か

GitHub の PR コメントは3つの異なる場所に分散している。1箇所しか見ないとインラインレビューなどが取りこぼされる。このスキルは3箇所全てを確認することで漏れを防ぐ。

## 実行フロー

### Step 1: 対象PRの特定

- ユーザーが PR 番号を指定した場合（例: 「PR #123 のレビューまとめて」）はその番号を使用する
- 指定がなければ、現在のブランチに紐づくPRを `gh pr view` で特定する
- PR番号とリポジトリ情報（owner/repo）を取得する

### Step 2: 全コメントの取得

以下の3つのエンドポイントを全て確認する。1つでも省略しない。3つは独立しているため並列で取得する。

1. **通常コメント**（PR ページ上部の会話コメント）
   - `gh api repos/{owner}/{repo}/issues/{pr_number}/comments`
   - CodeRabbit のサマリレビューはここに投稿される

2. **インラインレビューコメント**（コードの特定行に付くコメント）
   - `gh api repos/{owner}/{repo}/pulls/{pr_number}/comments`
   - CodeRabbit の個別の指摘やコード修正提案はここに投稿される

3. **レビューサマリ**（Approve / Request changes の本文）
   - `gh api repos/{owner}/{repo}/pulls/{pr_number}/reviews`
   - レビュー全体のまとめコメントはここに投稿される

### Step 3: 指摘事項の整理

3つのエンドポイント全てでコメントが0件の場合、レビューがまだ完了していない可能性がある。ユーザーに「まだレビューコメントが付いていません。少し待ってから再度実行してください」と通知して終了する。

コメントがある場合、レビュアーごと（CodeRabbit、code-review プラグイン、人間など）に分類し、以下の形式でまとめる:

- **指摘の種類**（バグ、パフォーマンス、スタイル、セキュリティなど）
- **対象ファイル・行**（インラインコメントの場合）
- **指摘内容の要約**
- **提案された修正方針**

### Step 4: 修正方針の提案

全ての指摘に対して、以下を提案する:

- 対応すべきもの / 対応不要なもの（理由付き）
- 対応する場合の具体的な修正方針
- 優先度（高：バグ・セキュリティ、中：パフォーマンス・設計、低：スタイル・好み）
