# ship

コード品質チェック（simplify）→ ブランチ作成 → コミット → push → PR作成 → code-review を一気通貫で実行する。

## 実行フロー

以下のステップを順番に実行する。各ステップで問題が発生した場合はユーザーに報告して判断を仰ぐ。

### Step 0: コード品質チェック（simplify）

ユーザーがすでに simplify を実行済みかどうかを確認する。

- **実行済みと明示された場合**: 変更差分をざっと確認し、明らかな問題がなければすぐ Step 1 へ進む
- **未実行の場合**: `/simplify` を実行する
  - 指摘事項がなければ Step 1 へ進む
  - 指摘事項がある場合、内容をユーザーに提示して確認を取る
    - 修正を承認 → 修正を適用してから Step 1 へ
    - スキップを選択 → そのまま Step 1 へ

### Step 1: ブランチ作成（main ブランチ上の場合のみ）

`git branch --show-current` で現在のブランチを確認する。

- **すでに feature ブランチにいる場合** → Step 2 へスキップ
- **main ブランチにいる場合**:
  1. 変更内容からブランチ名の案（feat/xxx, fix/xxx, docs/xxx, chore/xxx 形式）を提示してユーザーに確認を取る
  2. 承認されたら `git checkout -b <branch>` でブランチを作成

### Step 2: コミット（未コミットの変更がある場合のみ）

`git status` で未コミットの変更を確認する。

- **変更がない** → Step 3 へスキップ
- **変更がある場合**:
  1. 変更内容を確認してコミットメッセージの案を提示し、ユーザーに確認を取る
  2. 承認されたら `git add` でステージングして `git commit` を実行

### Step 3: push

main との差分が混入しないよう、push 前に最新化を行う。
特に worktree で作業している場合、元にした main が古いとマージ済みの変更が PR の差分に混入するため、rebase による最新化が重要になる。

1. `git fetch origin` を実行
2. `origin/main` が自分のブランチの起点より進んでいるか確認する
   - 進んでいる → `git rebase origin/main` を実行
     - 競合が出たらユーザーに報告し、解決するか `git rebase --abort` で中断するか判断を仰ぐ
   - 進んでいない → rebase しない
3. `git push` を実行（既にリモートに push 済みの場合は `--force-with-lease` を付ける）

### Step 4: PR作成

1. `.github/pull_request_template.md` のテンプレートを使って PR を作成する
2. コミットの内容からテンプレートの各セクションを埋める
3. `gh pr create` で PR を作成する

### Step 5: code-review

PR作成後、code-review プラグインを実行してレビューを受ける。

## 注意事項

- コミットが一切ない場合はユーザーに通知して終了する
